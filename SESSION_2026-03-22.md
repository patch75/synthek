# Session 2026-03-22 — Granulométrie + Refonte page Projet

## Résumé utilisateur

Nouvelle fonctionnalité opérationnelle : **import automatique de la granulométrie** depuis le fichier Excel architecte (plan masse / tableau de surface), avec restitution d'un tableau détaillé par bâtiment.

### Ce que vous pouvez faire maintenant

1. Ouvrir un projet → section **🏢 Bâtiments — Granulométrie**
2. Cliquer **📥 Importer Excel** → sélectionner le fichier architecte (.xlsx)
3. Vérifier la **proposition de regroupement** (tableau Montée → Bâtiment, modifiable)
4. Cliquer **✓ Confirmer et importer**
5. Le tableau synthèse s'affiche et est **persisté** — il réapparaît au rechargement

### Tableau synthèse D1

| Bâtiment | Montées | Logements | LLI | LLS | BRS | Acc.std | Acc.premium | Villas | Fiabilité |
|---|---|---|---|---|---|---|---|---|---|
| A | A1, A2 | 9 | 0 | 0 | 1 | 6 | 2 | 0 | haute |
| ... | | | | | | | | | |
| **TOTAL** | | **49** | | | | | | | |

### Règles métier appliquées automatiquement

- **PREMIUM** = logement au dernier niveau de sa montée, sans annotation sociale
- **VILLA** = logement dans un bloc nommé VILLA * (type produit distinct)
- **Financement** détecté via annotations (LLI) (LLS) (BRS) dans les N° de logement
- **Financement montée global** : ex. "C (LLS)" → tous logements sans annotation = LLS
- Surfaces m² ignorées, formules Excel non résolues ignorées

## Session 2026-03-22 (suite) — Refonte pipeline LLM universel

### Contexte

Le pipeline granulométrie déterministe (format-spécifique) a été remplacé par un pipeline LLM universel. L'ancien code n'extrayait pas correctement les bâtiments, villas, et financements sur des fichiers à structure variable.

### Nouvelle architecture pipeline

```
Frontend (Projet.jsx)
  └─ POST /projets/:id/granulometrie/proposer  (fichier base64 + nom_feuille optionnel)
       └─ Backend (projets.js)
            └─ POST http://127.0.0.1:5001/granulometrie/proposer
                 └─ extraire_granulometrie(bytes, nom, regroupement_valide=None)
                      ├─ _extraire_texte_brut_excel()  → texte positionnel F:f L:l C:c = val
                      ├─ _compter_logements_individuels()  → (count=49, tous_nos=[...])
                      └─ _appeler_sonnet()  → JSON avec nos_comptes par bâtiment
                           └─ _verifier_et_corriger_batiments()  → 3 règles anti-hallucination

Frontend — après validation utilisateur
  └─ POST /projets/:id/granulometrie/import  (fichier + regroupement = liste batiments validés)
       └─ extraire_granulometrie(bytes, nom, regroupement_valide=[...])
            └─ Appel 2 direct (pas de re-extraction Sonnet) → JSON D1 final
```

### Fonctions clés `extraire_granulometrie.py`

| Fonction | Rôle |
|---|---|
| `_extraire_texte_brut_excel()` | openpyxl read_only → texte positionnel, filtre float/dates |
| `_compter_logements_individuels()` | Compte N°s : lignes C:A="N°" + fallback ≥3 N°s, villas après VILLAS |
| `_formater_contrainte_nos()` | Formate liste exhaustive Python → contrainte prompt Sonnet |
| `_appeler_sonnet()` | claude-sonnet-4-6, max_tokens=4000, fallback extraction JSON |
| `_verifier_et_corriger_batiments()` | 3 règles vérification + dérivation financements annotations |
| `extraire_granulometrie()` | Point d'entrée : mode sélection feuille / validation / confirmation |

### Anti-hallucination : `nos_comptes`

Sonnet doit lister explicitement les N°s lus par bâtiment. Python vérifie :

1. `nb_logements == len(nos_comptes)` → corrigé automatiquement
2. Aucun N° en doublon inter-bâtiments → `nb_logements = null` si doublon
3. `sum(len(nos_comptes)) == nb_python` → tous à null si écart total

Financements dérivés des annotations N° : `(BRS)`, `(LLS)`, `(LLI)`, entiers 1-2 chiffres = villas, sans annotation = accession libre.

### Format JSON D1 (mis à jour)

```json
{
  "nom": "A1",
  "montees": [],
  "nos_comptes": ["001", "101 (BRS)"],
  "nb_logements": 2,
  "LLI": null, "LLS": null, "BRS": 1,
  "acces_std": 1, "acces_premium": null, "villas": 0,
  "fiabilite": "haute",
  "section_cctp": null, "feuilles_dpgf": []
}
```

### Routes parser Python (mises à jour)

```
POST /granulometrie/proposer  { fichier, nom_fichier, nom_feuille? }
  → { etape: "selection_feuille", feuilles_disponibles, feuille_suggeree }
  ou { etape: "validation", batiments: [...], total_logements: 49 }

POST /granulometrie/import    { fichier, nom_fichier, nom_feuille?, regroupement: [...] }
  → { batiments: [...], total_logements: 49, donnees_manquantes: [...] }
```

### Modifications UI Projet.jsx

- `importGranuloStep === 0` → condition affichage liste bâtiments (was `!== 2`)
- `null → "?"` (was `"-"`) dans tableau D1
- Bouton ✏️ Modifier après confirmation (step 2)
- Sélection feuille si multi-onglets éligibles
- `onClick={e => e.target.value = ''}` sur input file (rechargement même fichier)
- Message "⏳ Analyse IA..." pendant appel Sonnet

### Décisions d'architecture

- **Python = extraction mécanique** (N°s, totaux) — fiable, format-agnostique
- **Sonnet = attribution sémantique** (bâtiments, financements, montées) — universel
- **Résultat fiable ou null** — jamais afficher une valeur incertaine (données contractuelles BET)
- **Pas de parsing sémantique Python** (détection noms bâtiments dans merged cells) → fragile

## Session 2026-03-22 (suite 2) — Corrections `_verifier_et_corriger_batiments` + debug

### Corrections appliquées sur `_verifier_et_corriger_batiments`

#### 1. Suppression R2/R3
R2 (doublon inter-bâtiments) et R3 (total global) supprimées — les N° 001/101/201 se répètent légitimement dans chaque bâtiment (RDC/R+1/R+2 de chaque montée). Remplacées par un commentaire explicatif.

#### 2. Financement global depuis nom ou montées
Détection `(LLS)/(LLI)/(BRS)` dans `nom` ET `montees`. Si trouvé, tous les N°s libres (sans annotation individuelle) de ce bâtiment basculent dans ce financement.

#### 3. Warning [NOS-PREM] std/premium
Quand `std+premium ≠ libres` : warning explicite ajouté à `donnees_manquantes` + tout mis en `acces_std`.

### Bug identifié et corrigé — écrasement des financements Sonnet

**Bug :** `_verifier_et_corriger_batiments` recalculait les financements depuis les annotations individuelles dans `nos_comptes`, écrasant la bonne réponse de Sonnet. Exemple : C annoté `(LLS)` en entête → Sonnet retourne `LLS=8` (correct) → Python trouve 0 fois `(LLS)` dans les N°s individuels → écrase avec `LLS=null, acces_std=8`.

**Diagnostic :** print debug JSON Sonnet brut → confirmé `LLS=8` correct de Sonnet. Print debug BODY /import → confirmé que le frontend renvoyait déjà `LLS=null, acces_std=8` (corrompu à l'étape /proposer).

**Fix appliqué :**
```python
somme_sonnet = LLI + LLS + BRS + acces_std + acces_premium + villas
if somme_sonnet == nb_logements:
    # Sonnet cohérent → conserver sa répartition
else:
    # Sonnet incohérent → dériver depuis annotations nos_comptes (fallback)
    # + financement global depuis nom/montées
    # + warning [FIN-DERIVE]
```
