### PATCH_GRANULOMETRIE appliqué

#### MOD 1 — backend/src/routes/projets.js
4 nouvelles routes granulométrie :
- `POST /projets/:id/granulometrie` — import JSON ou Excel, normalisation, stockage
- `GET /projets/:id/granulometrie` — lecture + mapping auto calculé
- `PATCH /projets/:id/granulometrie/valider` — active le diff Python
- `PATCH /projets/:id/granulometrie/invalider` — désactive pour modification

**Principe** : une feuille DPGF = un bâtiment. Le code regroupe automatiquement par `section_cctp` :
```
BAT A (§3) + BAT B (§3) → CCTP_section_3: ["BAT A", "BAT B"]
BAT C (§4)               → CCTP_section_4: ["BAT C"]
BAT D (§5) + BAT E (§5)  → CCTP_section_5: ["BAT D", "BAT E"]
VILLA 123 (§6) + VILLA4  → CCTP_section_6: ["VILLA 123", "VILLA4"]
```

## Workflow d'utilisation

1. Ouvrir un projet → section "🗺 Granulométrie" → "+ Configurer"
2. Coller le JSON `granulometrie_allegorie.json` dans le champ Import JSON → Importer
3. Vérifier le mapping calculé automatiquement
4. "Enregistrer" → "✓ Valider"
5. Relancer la comparaison DPGF vs CCTP
6. Le diff Python est maintenant actif avec le bon mapping → 0 faux positif SECTION_X

## Problèmes connus / Limites

### 1. Types de logements non saisis dans l'UI
Le modal ne demande pas `types_logements` ni `systeme_chauffage` / `systeme_vmc`.
Ces champs existent dans le JSON et la BDD mais ne sont pas exposés dans le formulaire v1.
→ Contournement : importer directement le JSON complet (granulometrie_allegorie.json inclut tout).

### 2. Faux positifs Python restants
- **"Attente évier" vs "Évier"** : Python génère un C02 car ce sont deux prestations distinctes selon le type de logement (accession vs social). Python ne connaît pas le sous-périmètre par type de logement.
  → Décision : marquer ✗ Faux positif dans la méthode légère. Option C (ne rien toucher au Python).
- **"Attente" plomberie** : peut être capturé par `est_ligne_exclue()` si nécessaire.

### 3. Sonnet forcé
Le ternaire Haiku/Sonnet a été supprimé → toutes les comparaisons utilisent Sonnet.
Le switch de modèle dans le modal "Relancer comparaison" est maintenant inactif côté backend (mais reste dans l'UI).
→ À clarifier : supprimer le switch UI ou restaurer le ternaire.

### 4. Pause 2s / fast path
Avec le fast path actif (mapping validé), la pause entre sections n'existe plus — 1 seul appel.
Fallback section par section : pause 2s (à surveiller si 429 Sonnet).

## Fast path — Appel unique Claude (ajout fin de session)

### Architecture
```
mapping validé + DPGF + mode technique
        ↓
Python → N écarts groupés par bâtiment
        ↓
1 appel Claude Sonnet (extraits Python uniquement, pas le texte brut)
        ↓
Claude valide chaque écart (T1-T8) → alertes BDD
```

### Gain de temps
```
AVANT (7 sections) : 7 × 20s + 6 × 2s = ~152s
APRÈS (fast path)  : 1 × 25s           = ~25s
```

### Comportements
- **Python = 0 écart** → retour immédiat CONFORME, 0 appel Claude
- **Python = N écarts** → 1 appel Claude avec tous les extraits groupés par bâtiment
- **Mapping non validé** → fallback ancien mode (section par section)
- **Mode chiffrage** → fallback (fast path désactivé, chiffrage = hors périmètre Python)

### Fichier modifié
`backend/src/services/comparerDocuments.js` :
- Nouvelle fonction `appelerClaudeSingleCall()` (~90 lignes)
- Fast path inséré après nettoyage des anciennes alertes, avant la boucle sections
